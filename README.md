# The Gaming Room
Software document and prototype for the web based game application Draw It or Lose it

**Briefly summarize The Gaming Room client and their software requirements. Who was the client? What type of software did they want you to design?**
-The client was a company named The Gaming Room, which required a web based version of their game Draw It or Lose It. This application needed to work across multiple platforms and users while keeping track of game sessions, teams, and their massive image library. The goal was to design a scalable application using a client server architecture that could manage performance and user interaction efficiently.

**What did you do particularly well in developing this documentation?**
-One thing I did well was keeping things split up into sections, especially separating client side functionality from server side processing. That made it easier to understand data flow through the application and how the different parts of the system interact.

**What about the process of working through a design document did you find helpful when developing the code?**
-Creating the design document made me conceptualize how the application would work before actually coding. I had to plan out how requests would be handled, data would be stored, and how it would scale going forward. This made the development process smoother because I knew the direction I'm going for rather than jumping in head first.

**If you could choose one part of your work on these documents to revise, what would you pick? How would you improve it?**
-I would change the requirements section to expand on edge cases and user interaction. Although it was explained what needed to be done, I believe by specifying certain situations it would strengthen your design and prevent confusion while writing code. Additionally, it would allow your system to handle unexpected situations.

**How did you interpret the user’s needs and implement them into your software design? Why is it so important to consider the user’s needs when designing?**
-Performance, scalability, and ease of use were factors I considered when understanding the user’s needs. Since the game relies on quick interaction, performance was essential. It also had to perform those tasks while multiple users were using the system. Taking into consideration what the user needs is crucial when developing an application. If you do not meet the users needs, they will not have an application that functions well or usable.

**How did you approach designing software? What techniques or strategies would you use in the future to analyze and design a similar software application?**
-I tackled the problem by dividing up the system into parts and examining how they communicate with one another. I employed modular design as well as organized around client-server principles in order to maintain separation of concerns. If I had more time, I would design in the same way and spend more time solidifying requirements up front.
